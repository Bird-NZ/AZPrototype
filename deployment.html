AZ Prototype Docs

Deployment, dry-run, and rollout control

How to validate and deploy safely with staged execution, plan previews, and rollback awareness.

Dry-run first

az prototype deploy --dry-run

That uses Terraform plan or Azure deployment What-If depending on the generated infrastructure path.

Deploy for real

az prototype deploy
az prototype deploy --stage 2

Stage targeting is useful when iterating on a single part of the generated system.

Interactive deploy session commands

/status
/plan N
/deploy N
/rollback N
/redeploy N
/outputs
/preflight

These session commands matter in demos and live troubleshooting because they keep the operator inside one guided flow.

What happens before deployment

  • Subscription and project checks
  • Resource provider validation
  • Resource group or deployment context setup
  • Plan preview for infra stages
  • Output capture for downstream stages

Deployment habits worth enforcing

  1. Run dry-run before every customer-facing or shared-environment deployment.
  2. Inspect names, SKUs, and security-sensitive defaults before approval.
  3. Prefer incremental iteration by stage over giant all-at-once retries.
  4. Keep evidence: plan output, deployment output, and resulting URLs.
