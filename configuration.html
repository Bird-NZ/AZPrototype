AZ Prototype Docs

Configuration and prototype.yaml

How project configuration works, which settings matter most, and how naming stays consistent.

Example configuration

project:
  name: my-prototype
  location: eastus
  environment: dev
  iac_tool: terraform

naming:
  strategy: microsoft-alz
  org: contoso
  env: dev
  zone_id: zd

ai:
  provider: copilot
  model: claude-sonnet-4

agents:
  custom_dir: ./.prototype/agents/
  overrides: {}

deploy:
  track_changes: true

Key sections

  • project: identity, environment, region, IaC tool
  • naming: naming strategy and org/environment markers
  • ai: provider and model routing
  • agents: custom agent path and overrides
  • deploy: deployment behavior such as change tracking

Naming strategies

  • microsoft-alz — default landing zone-style naming
  • microsoft-caf — Cloud Adoption Framework inspired pattern
  • simple — lightweight names for quick prototypes
  • enterprise — broader enterprise pattern with business-unit shape
  • custom — your own convention

Configuration commands

az prototype config init
az prototype config show
az prototype config get --key project.location
az prototype config set --key ai.provider --value azure-openai

Operator advice

Lock down naming early. Renaming conventions midstream is the kind of “small change” that quietly turns into broken references, inconsistent outputs, or a messy demo.
