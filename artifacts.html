Use this when teaching: Most teams do not start with a perfect spec. They start with a brief, messy notes, screenshots, spreadsheets, and a handful of constraints. That is exactly what --artifacts is for.

How to teach this well

  1. Show the files first. Let people see the folder and file types.
  2. Explain the point. Real projects start from mixed artifacts, not one pristine prompt.
  3. Run the command. Use design --artifacts with a tight context sentence.
  4. Narrate what the system is doing. Biz analyst closes gaps; architect turns context into design.

What to avoid

  • Too many duplicate files
  • Noisy exports without signal
  • Artifacts with no clear problem statement
  • Skipping the context sentence entirely
