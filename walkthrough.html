AZ Prototype Docs

Init → Design → Build → Deploy walkthrough

A full tutorial that explains the mental model of each stage, not just the commands.

Stage 1 — Init

Goal: establish project identity, configuration, AI provider, environment, and optional workload template.

az prototype init   --name retail-demo   --location australiaeast   --iac-tool terraform   --ai-provider azure-openai   --environment dev   --template ai-app

Use templates as accelerators, not cages. They provide a sensible starting service topology.

Stage 2 — Design

Goal: turn requirements into an architecture and clarify gaps.

az prototype design --artifacts ./requirements --context "Customer support copilot with conversation history and secure APIs"
  • The business analyst looks for missing requirements.
  • The cloud architect turns requirements into an architecture plan.
  • You can rerun design to refine decisions instead of starting over.

For demos and training, use the included sample artifact pack to show what --artifacts really means in practice.

mkdir -p requirements
cp sample-artifacts/* ./requirements/
az prototype design --artifacts ./requirements --context "Use these mixed-format inputs to create a pragmatic Azure prototype design."

That sample pack includes a brief, discovery notes, slide outline, CSV issue list, JSON non-functional requirements, a PDF brief, and a screenshot mock.

Stage 3 — Build

Goal: derive a staged deployment plan and generate the implementation assets.

az prototype build

Build is where the system selects matching templates, derives a deployment plan, assigns specialized agents, and generates staged outputs for infra, docs, and apps.

  • Fine-grained stages keep dependencies explicit.
  • Governance policies and anti-pattern checks are applied during generation.
  • QA review can route issues back to generators before you proceed.

Stage 4 — Deploy

Goal: validate and roll out the generated stages into Azure.

az prototype deploy --dry-run
az prototype deploy

Deploy supports staged rollout, preflight checks, outputs capture, slash-command controls in interactive sessions, and rollback flows.

Re-entrant stages

design, build, and deploy are designed to be revisited. That matters in real work: requirements change, QA finds issues, or a customer asks to add Redis halfway through.

Operator habit to keep

Before every meaningful deploy, run a dry-run, inspect the stage plan, and confirm the chosen naming strategy and environment are still correct.
